# การติดตั้งและใช้งานระบบบันทึกประวัติการแก้ไข (Modification History)

## 📋 สรุปการเปลี่ยนแปลง

ระบบ Dashboard ได้รับการปรับปรุงให้สามารถแสดงประวัติการแก้ไขที่สมบูรณ์แบบ แทนที่จะแสดงเพียงการแก้ไขล่าสุดเท่านั้น

### ✨ คุณสมบัติใหม่

1. **ประวัติการแก้ไข (Modification History)**
   - แสดงทุกการเปลี่ยนแปลงของคำสั่งพิมพ์
   - ระบุวันเวลาที่แก้ไข ⏰
   - ระบุว่าใครแก้ไข (ชื่อและรหัสพนักงาน)
   - แสดงสิ่งที่เปลี่ยนแปลง (สรุปรายละเอียด)

2. **การแสดงผลแบบ Expandable**
   - ปุ่มขยาย/ยุบเพื่อดูประวัติทั้งหมด
   - แสดงการแก้ไขล่าสุดด้านบนสุด
   - เลื่อนดูได้สำหรับประวัติที่นานๆ

3. **การบันทึกอัตโนมัติ**
   - บันทึกเหล่านี้จะทำงานได้ทั้งกับระบบเก่า (updated_at, updated_by) และระบบใหม่ (audit_logs table)

## 📁 ไฟล์ที่เพิ่มหรือแก้ไข

### ไฟล์ใหม่ที่เพิ่ม:

1. **`src/app/api/orders/[id]/history/route.ts`**
   - API endpoint สำหรับดึงประวัติการแก้ไข
   - สนับสนุนการดึงจาก audit_logs ตาราง
   - มี fallback สำหรับระบบเก่า

2. **`src/app/printer/components/EditHistory.tsx`**
   - React component แสดงประวัติการแก้ไข
   - รองรับการขยาย/ยุบได้
   - โหลดข้อมูลแบบ lazy loading

3. **`setup_enhanced_audit_logs.sql`**
   - Script สำหรับสร้าง audit_logs table
   - สร้าง trigger เพื่อบันทึกการเปลี่ยนแปลงโดยอัตโนมัติ
   - สร้าง index สำหรับประสิทธิภาพ

### ไฟล์ที่แก้ไข:

1. **`src/app/printer/dashboard/page.tsx`**
   - เพิ่ม import EditHistory component
   - แทนที่การแสดงผลการแก้ไขเก่า ด้วย EditHistory component
   - เพิ่มฟังก์ชัน `logAuditTrail()` สำหรับบันทึกการเปลี่ยนแปลง
   - เพิ่มการเรียก logAuditTrail ใน:
     - `saveEdit()` - เมื่อแก้ไขข้อมูล
     - `verifyOrder()` - เมื่อตรวจสอบ
     - `handleCancelOrder()` - เมื่อยกเลิก

ถ้าต้องการ tracking ที่สมบูรณ์ยิ่งขึ้น ให้รันไฟล์ SQL นี้ใน Supabase Console:

```sql
-- ดูที่ไฟล์: setup_enhanced_audit_logs.sql

```

**หมายเหตุ:** 
- ถ้า audit_logs table ยังไม่มี script นี้จะสร้างให้
- ถ้าต้องการให้ trigger บันทึกการเปลี่ยนแปลงโดยอัตโนมัติ ต้องรันสคริปต์นี้

### ขั้นที่ 2: ทดสอบระบบ

1. ไปที่หน้า Dashboard
2. แก้ไขข้อมูลคำสั่งพิมพ์เสก
3. คลิกปุ่ม "ประวัติการแก้ไข" เพื่อขยายดูประวัติ
4. ควรเห็นรายการการแก้ไขล่าสุด พร้อมชื่อผู้แก้ไขและสิ่งที่เปลี่ยนแปลง

## 🔒 ข้อมูลความปลอดภัย

- ระบบบันทึกเก็บบันทึกการเปลี่ยนแปลงเท่านั้น
- ไม่มีการลบ update ประวัติ (immutable)
- สามารถอ่านได้โดยผู้ใช้ทั้งหมด
- RLS Policy ถูกตั้งค่าให้อ่านได้เท่านั้น

## ⚙️ การกำหนดค่า (Optional)

ถ้าต้องการเปลี่ยน cleanup policy ของ audit logs:

แก้ไข `setup_audit_log_cleanup.sql`:
```sql
-- ปัจจุบัน: ลบข้อมูล audit_logs ที่เก่ากว่า 7 วัน ทุกวันเที่ยงคืน
SELECT cron.schedule(
  'delete_old_audit_logs',
  '0 0 * * *',  -- ทุกวัน 00:00 น.
  $$ DELETE FROM public.audit_logs WHERE created_at < NOW() - INTERVAL '7 days'; $$
);

-- เปลี่ยนเป็นเก็บนานขึ้น (30 วัน):
-- INTERVAL '7 days' → INTERVAL '30 days'
```

## 🐛 Troubleshooting

### ปัญหา: ไม่เห็นประวัติการแก้ไข

**สาเหตุ:** 
- ยังไม่มีการแก้ไขคำสั่ง
- audit_logs table ยังไม่ถูกสร้าง

**วิธีแก้:**
1. ตรวจสอบว่ามีการแก้ไขคำสั่งหรือไม่
2. รัน setup_enhanced_audit_logs.sql ใน Supabase
3. ล้าง cache (Ctrl+Shift+Delete)

### ปัญหา: API return 404

**สาเหตุ:** 
- Order ID ไม่ถูกต้อง
- ไม่พบคำสั่งนี้

**วิธีแก้:**
1. ตรวจสอบ Order ID ว่าถูกต้องหรือไม่
2. ตรวจสอบสิทธิ์ RLS policy
